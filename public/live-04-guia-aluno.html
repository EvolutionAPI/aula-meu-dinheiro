// Pipeline EVO — Etapas 1 a 10

Etapa 1

Inicialização do projeto local + Git

Base limpa antes do framework. Faça isso antes de qualquer instalação.

Criação da pasta + git remoto bash 
mkdir meu-dinheiro
cd meu-dinheiro

git init
git branch -M main
git remote add origin git@github.com:EvolutionAPI/aula-meu-dinheiro.git
  • Repo deve ser privado para live de construção.
  • Se der erro de remote duplicado: git remote remove origin e refazer.
  • Validar acesso SSH antes de começar: ssh -T git@github.com.
Etapa 2

Instalação do EVO Method

Evitar configuração inchada: só o que será usado.

Instalador bash 
npx @evoapi/evo-method install
  1. Diretório: aceitar a pasta do projeto
  2. Módulos: EVO Core + módulo Agile-AI
  3. Integrações: marque apenas ferramentas reais da live
  4. Idioma do agente e output docs: Português
  5. Output: manter padrão _evo-output/
Não marcar módulos que não serão usados — gera ruído nos comandos e artefatos desnecessários.
Etapa 3

Brainstorming orientado

Primeiro artefato que alimenta o restante da pipeline.

Início da sessão terminal 
claude
/evo-brainstorming

Responder 3 coisas com objetividade:

  • O que vai criar
  • Resultado esperado
  • Público alvo
Se travar, use técnica explícita: What If, SCAMPER, Analogical, Reverse Brainstorming. Diga ao agente qual usar.
Etapa 4

Product Brief com active_feature definido

Sem feature ativa, artefatos ficam bagunçados.

Workflow command 
/evo-create-product-brief
  • Quando pedir slug, usar meu-dinheiro.
  • Isso alimenta o active_feature no config.
  • Confirme pasta correta de saída antes de continuar.
Se a feature não estiver ativa command 
/set-feature meu-dinheiro
Etapa 5

PRD completo

Documento de execução, não texto bonito. Requisitos precisam ser testáveis.

Workflow command 
/evo-create-prd

Obrigatório no PRD

  • Resumo executivo com escopo real
  • Critérios de sucesso mensuráveis
  • MVP Strategy + out of scope
  • Jornadas primárias + edge case

Qualidade mínima

  • Não pode ser genérico — se valer para qualquer app, está errado
  • Cada FR deve ser testável
  • Escopo deve caber na live
Etapas 6 e 7

UX Design + Architecture

Primeiro experiência, depois implementação técnica. Nessa ordem.

Comandos command 
/evo-create-ux-design
/evo-create-architecture

UX Design deve fechar

  • Interação principal ("signature interaction")
  • Sistema visual + acessibilidade
  • Jornadas com pontos de fricção
  • Estados: vazio, erro, carregando

Architecture deve incluir

  • Stack definitiva e justificada
  • Schema de banco validado
  • Plano de deploy (Vercel + env vars)
  • Decisões explícitas (ADRs)
UX e Architecture devem estar alinhados entre si antes de gerar epics. Desalinhamento aqui gera retrabalho nas stories.
Etapas 8 e 9

Epics / Stories + Readiness + Sprint

Transformar documento em trabalho executável.

Comandos command 
/evo-create-epics-and-stories
/evo-check-implementation-readiness
/evo-sprint-planning
  • Validar cobertura completa FR → Epic (ninguém fica órfão).
  • Readiness precisa encontrar PRD + Architecture + UX + Epics.
  • Sprint status vira fonte de verdade do estado da live.
Se o readiness apontar gaps, volte e complete o documento faltante — não avance com lacunas.
Etapa 10

Loop de execução por Story

Cria, desenvolve, revisa, commita, sobe. Repete.

Loop operacional command 
/evo-create-story 1.1
/evo-dev-story 1.1
/evo-code-review 1.1

git add .
git commit -m "feat(epic-1): story 1.1 - schema inicial + tema visual"
git push origin main

# repetir para 1.2, 1.3, ...
# ao finalizar épico
/evo-retrospective epic 1
  • Não abrir 3 stories em paralelo — uma de cada vez.
  • Code review antes do commit, sem exceção.
  • Push imediato após commit — nunca acumular.
  • Retrospective ao fechar cada épico, não apenas no final.

// Deploy — 5 passos para produção

Deploy 1/5

Criar banco PostgreSQL no Supabase

SQLite não funciona na Vercel (filesystem efêmero). Supabase oferece PostgreSQL gratuito.

  1. Acessar supabase.com e criar conta (login com GitHub)
  2. Clicar New Project, escolher nome e senha do banco, região South America (São Paulo)
  3. Aguardar criação do projeto (~1 min)
  4. Ir em Settings → Database
  5. Copiar a Connection string → URI (Transaction mode / porta 6543)
.env env 
# Substitua pelos valores do Supabase Dashboard > Settings > Database > Connection string
DATABASE_URL="postgresql://postgres.SEU-PROJECT-REF:SUA-SENHA@aws-0-sa-east-1.pooler.supabase.com:6543/postgres?pgbouncer=true"
JWT_SECRET="gere-uma-string-segura-com-32-chars-minimo"
A senha do banco é definida na criação do projeto. Se esquecer, pode resetar em Settings → Database → Reset database password.
Deploy 2/5

Criar tabelas no Supabase

Prisma cria as tabelas automaticamente com um comando.

Criar tabelas bash 
# Com o .env configurado com a URL do Supabase:
npx prisma db push
  • Isso cria as tabelas User, Category e Transaction no Supabase
  • Verificar no Supabase Dashboard → Table Editor se as tabelas apareceram
  • Testar localmente: npm run dev — cadastrar e logar deve funcionar
Se npx prisma db push rodou sem erro e as tabelas aparecem no Supabase, o banco está pronto.
Deploy 3/5

Preparar e importar na Vercel

Build local + deploy contínuo via Git.

Validar build local bash 
npm run build   # deve compilar sem erros
  1. Vercel Dashboard → Add New → Project
  2. Conectar GitHub e selecionar o repositório
  3. Revisar Framework Preset (Next.js deve ser detectado automaticamente)
  4. Adicionar Environment Variables antes do deploy:
    DATABASE_URL = connection string do Supabase
    JWT_SECRET = string segura com 32+ caracteres
  5. Clicar Deploy
As env vars devem ser adicionadas ANTES do primeiro deploy. Sem elas, o build vai falhar.
Deploy 4/5

Domínio customizado + DNS na Cloudflare

Copiar exatamente o que a Vercel pedir — sem adaptar.

  1. Vercel → Settings → Domains → Add Domain
  2. Adicionar seudominio.com e www.seudominio.com
  3. Copiar registros DNS do painel da Vercel
  4. Cloudflare → DNS → Add record com os valores exatos
  5. Aguardar verificação no painel da Vercel

Cenário comum

  • www como CNAME
  • @ como A record ou ALIAS
  • Proxy Cloudflare: DNS only (cinza, não laranja)

Se der problema

  • Verificar TTL — aguardar propagação (até 5 min)
  • Confirmar que Cloudflare proxy está desligado
  • Usar dig seudominio.com para verificar
Deploy 5/5

SSL, testes finais e deploy contínuo

A validação de produção fecha a live.

  • ✓ Testar https://seudominio.com
  • ✓ Testar https://www.seudominio.com
  • ✓ Criar conta e fazer login no app em produção
  • ✓ Registrar uma transação e verificar no dashboard
  • ✓ Verificar dados no Supabase Dashboard → Table Editor
  • ✓ Confirmar certificado SSL emitido
Após conexão Git: push em branch de produção atualiza produção; branches normais geram preview deploy automático.
Troubleshooting

Falhas comuns e correção rápida

Use isso ao vivo quando der ruim.

Git / setup

  • Remote errado: git remote remove origin e recriar
  • Branch divergente: alinhar main com git pull --rebase
  • Push negado: validar auth SSH com ssh -T git@github.com

EVO workflow

  • active_feature vazio: rodar /set-feature meu-dinheiro
  • Doc não encontrado: validar paths no config
  • Saída fora da pasta: revisar config do EVO

Build / Vercel

  • Build falha: rodar npm run build local e resolver antes
  • Env var faltando: adicionar no painel da Vercel e re-deploy
  • Root Directory errado: corrigir em Project Settings

Supabase / Banco

  • "Tenant or user not found": DATABASE_URL com project-ref errado
  • "Can't reach database": usar porta 5432 para prisma db push, porta 6543 para o app
  • "Foreign key constraint": cookie JWT antigo — limpar cookies no browser
  • Tabelas não criaram: rodar npx prisma db push novamente

DNS / domínio

  • Não verificou: confirmar proxy Cloudflare está em DNS only
  • SSL pendente: aguardar até 10 min após DNS propagar
  • Subdomínio errado: conferir exatamente o que a Vercel pediu
